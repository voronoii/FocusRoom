## Problem Statement

집에서 혼자 일하거나 공부할 때 집중이 안 되는 문제. 카페에서 다른 사람이 일하고 있으면 나도 집중이 되는 "카페 효과(body doubling)"를 가상 공간에서 재현하고 싶다.

핵심 니즈: **대화 없는 소속감** — 누군가 옆에 있다는 느낌은 주되, 대화나 교류 부담은 없는 상태.

## Demand Evidence

- **창업자 자신의 고통**: 직접 이 문제를 겪고 있어서 만드는 서비스
- **시장 행동 증거**:
  - "Study with me" 유튜브 영상: 수십만~수백만 조회수
  - 한국 스터디카페 시장: SpaceCloud 기준 120,000개 공간, 33조 원 거래
  - Focusmate: Trustpilot 4.9/5, 수백만 세션
  - 디스코드 무음 공부방: 자생적으로 만들어짐
- **아직 없는 것**: 한국 시장에 가상 body doubling 앱이 없음. 물리적 스터디카페 문화만 존재.

## Status Quo

현재 사람들이 쓰는 해결책과 한계:

| 해결책 | 한계 |
|--------|------|
| Focusmate ($9/월) | 1:1 매칭 + 카메라 필수 + 대화 강제 → 부담스러움 |
| Study with me 유튜브 | 일방향, 내 존재감 없음, 책임감 없음 |
| 디스코드 무음방 | 존재감 없고, accountability 없음 |
| 스터디카페 | 물리적 이동 필요, 비용 5-10만원/월 |
| Forest/Focus 앱 | 혼자 하는 타이머, "함께" 느낌 없음 |

**핵심 갭**: Focusmate(너무 무겁고 강제적) ↔ 유튜브(존재감 제로) 사이의 빈 공간. 대화 없이도 "내가 여기 있고, 다른 사람도 여기 있다"는 느낌을 주는 서비스가 없다.

## Target User & Narrowest Wedge

**첫 100명 타겟**:
1. 원격근무 직장인 — 재택근무 중 외로움/집중력 저하
2. 수험생 — 스터디카페 대신 집에서 공부하고 싶은 학생

**Narrowest wedge**:
단일 웹 페이지. 접속 → 할 일 + 시간 입력 → 아바타 그리드에서 다른 사람과 함께 타이머 돌리기 → 완료 or 타임아웃.

## Constraints

- 웹 + 모바일 동시 지원 (PWA)
- 대화/채팅 기능 없음 (의도적 제한). 다중 방 확장 시에도 채팅은 추가하지 않음 — 방 선택은 주제 태그로만.
- Cold start 문제 해결 필수 (빈 방 = 가치 0) → MVP 전략: 실제 유저 5명 미만일 때 NPC 아바타 8-12개를 표시 (랜덤 태스크 + 타이머). 5-9명일 때 NPC를 점진적으로 감소 (목표: 총 인원 12명 유지). 실제 유저 10명 이상이면 NPC 0개. NPC는 UI에서 구분 불가.
- 한국어 우선. i18n은 MVP에서 미구현 — 문자열을 컴포넌트에 직접 작성. 영어 지원 시 next-intl 도입 예정.

## Premises

1. "대화 없는 소속감"은 디지털 환경에서도 집중력 향상 효과가 있다
2. 익명 아바타 + 목표 공개가 카메라 1:1보다 부담이 적으면서도 충분한 accountability를 제공한다
3. 한국 시장에서 스터디카페의 가상 대체재로 포지셔닝 가능 (월 5-10만원 대비 경쟁력)
4. Cold start 문제는 NPC 아바타, 시간대 집중, 커뮤니티 기반 런칭으로 해결 가능 — SNS 홍보만으로는 부족
5. 강제 로그아웃 메커니즘은 유지하되, 랜덤 아바타 vs 지속적 아바타는 A/B 테스트로 결정

## Cross-Model Perspective

Codex cold read (verbatim):

> **Steelman:** The strongest version is a digital study cafe that makes work commitments public without making people social. Users enter a silent room, declare a concrete task and timebox, occupy a visible seat, and either finish or fail in front of ambient peers. That is a real gap between passive "Study With Me" and high-friction "Focusmate".
>
> **Key insight:** "If I exist as an avatar in a space, there's a sense of responsibility assigned to each person." — This means they should build a public commitment system, not a metaverse, not a chat community, and not a generic productivity app.
>
> **Challenged premise:** "Forced logout + random avatar mechanics drive repeat visits" is wrong. Random identity probably hurts habit formation, recognition, and re-entry. A/B test persistent avatar vs random.
>
> **48-hour prototype:** Next.js PWA, Supabase (anon auth + Realtime Presence), Vercel, DiceBear avatars, PostHog analytics. One shared room, avatar grid, 25/50/90 min timers, auto-expire + result screen. Skip everything else.

**Cross-model synthesis:**
- Claude와 Codex 모두 "공개 커밋 시스템"이 핵심이라는 데 일치
- Codex가 랜덤 아바타 전제를 도전 → 창업자가 A/B 테스트로 결정하기로 합의
- 기술 스택 제안이 구체적이고 실행 가능

## Approaches Considered

### Approach A: "디지털 스터디카페" — Minimal Viable (선택됨)
단일 웹 페이지에 아바타 그리드. 접속 → 목표 입력 → 타이머 시작 → 완료/타임아웃. 가상 세계 없이 순수한 커밋 보드.
- Effort: S (human: ~1주 / CC: ~5시간)
  - Supabase 셋업 + 익명 인증 + DB 스키마: 30분
  - Entry 화면 (할 일 입력 + 시간 선택): 20분
  - Room UI + 아바타 그리드 + 반응형: 60분
  - 타이머 로직 (클라이언트) + Edge Function (서버 만료): 40분
  - 결과 화면 + MyPage (최근 20개 기록): 30분
  - PWA manifest + 서비스워커 + 오프라인 배너: 25분
  - PostHog 이벤트 계측 (6개 이벤트): 25분
  - NPC 아바타 로직 (동적 스케일링): 25분
  - 배포 + E2E 테스트: 25분
- Risk: Low
- Pros: 최소 개발, 빠른 검증, PWA로 앱/웹 동시 대응
- Cons: "카페 느낌" 약할 수 있음, 차별화 요소 약해서 "타이머 앱"으로 보일 위험. Phase 2(2D 맵 + 이동 + 말풍선)로 진화하여 해결.

### Approach B: "가상 카페 공간" — Ideal Architecture
2D 탑뷰 카페 맵에서 아바타가 자리에 앉아있는 시각적 공간.
- Effort: M (human: ~3주 / CC: ~6시간)
- Risk: Medium
- Pros: "카페에서 함께 일하는" 느낌 시각적 전달, 차별화 명확
- Cons: 2D 맵 복잡도, 모바일 대응 어려움, 가설 검증 전 디자인에 시간 소모

### Approach C: "커밋 타임라인" — Creative/Lateral
공간 메타포를 버리고 실시간 피드. 현재 작업 중인 사람 목록이 흐르는 형태.
- Effort: S (human: ~1주 / CC: ~1.5시간)
- Risk: Low
- Pros: 모바일 최적, 친숙한 UX
- Cons: "함께 있는 느낌" 약함, SNS와 혼동 위험

## Recommended Approach

**Approach A: 디지털 스터디카페 (Minimal Viable)**

이유: 핵심 가설("공개 커밋 + 타이머 + 다른 사람의 존재 → 집중력 향상")을 가장 빠르게 검증. 검증 후 Approach B(카페 맵)로 진화 가능. Claude와 Codex 모두 동일한 결론.

**기술 스택 (Codex 제안 기반):**
- Next.js PWA
- Supabase (email auth + anonymous auth + Postgres + Realtime Presence)
- Vercel 배포
- @dicebear/core (클라이언트 사이드 아바타 생성)
- PostHog analytics

**인증 & 세션 관리:**
- **이중 인증 모델**: 게스트(익명) + 이메일 가입 선택
  - 게스트: Supabase anonymous auth → 즉시 이용 가능, 기록은 브라우저 세션에 종속
  - 이메일: Supabase email auth (비밀번호 + 이메일 확인 링크) → 기록 영구 보존
- Auth 화면: 게스트 우선 레이아웃 — "바로 시작하기 (게스트)" primary CTA, "이메일로 시작하기" secondary
- 게스트 → 이메일 전환: Result 화면 또는 MyPage에서 "기록을 저장하려면 이메일 등록" 유도
- 로그인 유저 재방문: localStorage 토큰 확인 → 자동 로그인 → Entry 화면 직행
- 게스트 재방문: 쿠키 존재 시 자동 진행, 삭제 시 Auth 화면으로 돌아감 (기존 기록 소실)
- 활성 세션 재접속: 세션 감지 시 Room으로 바로 이동 + "세션 이어서 하기" 토스트 표시
- 탭 닫기 시: Realtime Presence에서 자동 제거 (Supabase 기본 동작). 세션 만료는 Supabase Edge Function (cron trigger, 1분 간격)으로 처리 — `sessions` 테이블에서 `started_at + duration > now()` 인 레코드를 `status = 'timeout'`으로 업데이트. 클라이언트 부재 시에도 타임아웃 기록 보장.

**화면 흐름 (4개 화면):**
```
첫 방문:    URL → Auth (게스트/이메일 선택) → Entry → Room → Result
재방문 (로그인): URL → (자동 로그인) → Entry
재방문 (게스트): URL → (쿠키 확인) → Entry (쿠키 없으면 Auth)
활성 세션:  URL → (세션 감지) → Room (바로 이동)
게스트→전환: Result/MyPage → 이메일 등록 → 기존 기록 이관
```

**Entry 화면 정보 위계:**
1. 접속자 수 ("●12명이 지금 집중 중") — 사회적 증거로 입장 동기 부여
2. 할 일 입력 필드 — 핵심 액션
3. 시간 선택 (25/50/90분) — 부차 선택
4. 입장하기 CTA — 아바타는 입장 후 Room에서 확인

**PWA 오프라인 동작:**
- 오프라인 시 타이머는 로컬에서 계속 동작
- 재연결 시 서버와 동기화: **서버 우선(server-wins)** 정책. 클라이언트가 "완료" 전송했지만 서버에서 이미 타임아웃 처리된 경우, 서버 상태(타임아웃)가 우선. 클라이언트가 타이머 만료 전에 재연결하면 서버 남은시간으로 동기화.
- 오프라인 상태에서는 다른 아바타가 보이지 않음 — "연결 끊김" 배너 표시

**타이머 동기화 아키텍처:**
- 타이머 카운트다운은 **클라이언트에서 계산** — DB에 저장된 `started_at` + `duration_minutes`로부터 남은 시간을 로컬 계산. Realtime으로 매초 동기화하지 않음.
- Supabase Realtime Presence는 입장/퇴장 이벤트에만 사용 (접속자 목록 갱신)
- 다른 유저의 남은 시간도 클라이언트가 해당 유저의 `started_at`과 `duration`으로 로컬 계산
- Edge Function (1분 간격 cron)은 서버 측 만료 처리 전용 — 최대 59초 오차 허용 (MVP)

**NPC 아바타 행동 규칙:**
- 타이머: 입장 시 랜덤 남은시간(10-70분) 부여, 클라이언트가 로컬 카운트다운
- 태스크 이름: 사전 정의된 풀(20개)에서 랜덤 선택 (예: "영어 공부", "코딩 연습", "보고서 작성")
- 세션 종료: 완료 70% / 타임아웃 30% 비율로 랜덤. 종료 시 퇴장 애니메이션 후 새 NPC 입장 (총 인원 유지)
- 입/퇴장: 5-15분 간격으로 랜덤하게 NPC 1명 퇴장 → 새 NPC 입장 (자연스러운 흐름 연출)

**방 용량:**
- 최대 30명 (NPC 포함). 30명 초과 시 신규 입장 대기열 — "잠시 후 다시 시도해주세요" 메시지 표시
- MVP에서 방은 1개. 30명 초과가 빈번해지면 Phase 2에서 다중 방 도입

**Supabase 스케일링:**
- 무료 티어: 200 동시 연결 → 초기 100 DAU에서는 충분 (동시 접속 ~20-30명 예상)
- DAU 500+ 시 Pro 플랜 ($25/월) 업그레이드 필요
- DAU 2000+ 시 커스텀 Realtime 솔루션 검토

**핵심 측정 지표 (PostHog 이벤트):**

| 이벤트 | 속성 | 트리거 |
|--------|------|--------|
| `session_start` | task_name, duration_minutes | 입장 버튼 클릭 |
| `room_joined` | user_count_at_join | 방 진입 완료 |
| `session_completed` | actual_duration_seconds | "완료" 버튼 클릭 |
| `session_timeout` | overtime_seconds | 타이머 만료 |
| `recommit` | gap_seconds (이전 세션 종료 후 경과) | 결과 화면에서 "다시 입장" 클릭, gap < 300초 |
| `next_day_return` | days_since_last | 24시간+ 경과 후 첫 세션 |

**PostHog 사용자 식별:**
- `distinct_id`로 Supabase `user.id` 사용 (게스트/이메일 모두 UUID)
- 이메일 유저: 영구 안정적 ID
- 게스트 유저: 쿠키/스토리지 삭제 시 새 `user.id` 생성 → 별도 유저로 카운트됨 (허용)
- 게스트 → 이메일 전환 시: PostHog `alias()` API로 게스트 ID → 이메일 계정 병합

**아바타 생성:**
- @dicebear/core 클라이언트 라이브러리 사용 (네트워크 의존성 없음)
- fallback: 이니셜 + 랜덤 배경색 아바타

**Room 화면 정보 위계:**
1. 내 타이머 바 (상단 sticky) — 아바타 + 태스크명 + 남은시간 + "완료" 버튼. 항상 보임.
2. 방 헤더 (타이틀 + 접속자 수) — 방 상태 확인
3. 아바타 그리드 (스크롤 영역) — 다른 사람들의 존재감. 30명이어도 내 타이머는 항상 상단에 고정.

**아바타 그리드 상세:**
- 동적 그리드: 3열, 행 수는 접속자 수에 따라 증가 (최대 30명 = 10행)
- 각 아바타: 아바타 이미지 (@dicebear/core, fallback: 이니셜+배경색) + 이름 + 태스크명 (최대 15자, 말줄임) + 남은시간 (5분 미만은 빨간색)
- 내 카드는 그리드에서 제외 (상단 타이머 바로 분리)
- **스타일: 테두리 없는 아바타** — 카드 border/shadow 없이 아바타+텍스트만 플로팅. "카페 자리에 앉은 사람들" 느낌. SaaS 대시보드 카드 그리드 패턴 회피.

**AI Slop 방지 디자인 원칙:**

## 반응형 & 접근성 (Responsive & Accessibility)

**뷰포트별 레이아웃:**
```
Mobile (< 480px):     그리드 2열, 타이머 바 하단 고정, Auth/Entry 전체폭
Tablet (480-768px):   그리드 3열, 좌우 패딩 24px
Desktop (768px+):     그리드 4열, max-width: 600px 중앙 정렬, 좌우 여백 auto
```

**터치 타겟:** 모든 버튼/탭 영역 최소 44×44px

**키보드 내비게이션:**
- Entry: Tab으로 할일 입력 → 시간 선택 → 입장 버튼 이동, Enter로 제출
- Room: Esc로 "완료" 확인 모달, Tab으로 사운드 토글 접근
- Auth: Tab으로 게스트/이메일 버튼 이동

**스크린 리더:**
- 타이머: `aria-live="polite"` (1분 간격 읽기), 5분 미만 시 `aria-live="assertive"`
- 접속자 수: `aria-label="현재 12명이 집중 중"`
- 아바타 그리드: `role="list"`, 각 아바타 `role="listitem"` + `aria-label="곰 #1092, TOEIC 단어 암기, 22분 10초 남음"`
- confetti: `aria-hidden="true"` (장식 요소)

**색상 대비:** 모든 텍스트 WCAG AA 충족 (4.5:1 이상). 밤 모드(`--bg-evening`)에서 텍스트는 `#E0E0E0` 사용.

**모션 감소:** `prefers-reduced-motion` 감지 시 — confetti/스태거드 입장 애니메이션 스킵, 즉시 전환.

## Phase 2 로드맵 (Approach B로 진화)

Approach A에서 핵심 가설이 검증되면 (D7 리텐션 15%+ 달성 시) Approach B로 진화:

1. **아바타 이동**: 2D 카페 맵에서 아바타가 자유롭게 돌아다닐 수 있음. 자리에 앉으면 타이머 시작.
2. **말풍선 리액션**: 다른 아바타에 호버 시 3개 버튼 표시 — [👋 인사] [🎉 축하] [💪 파이팅]. 클릭하면 상대 아바타 위에 말풍선이 3초간 표시. 대화는 없지만 최소한의 따뜻한 인터랙션.
3. **자리 선택**: 카페 맵에서 빈 자리를 클릭하여 자리 선택 가능.
4. **카페 테마**: 시간대별 카페 분위기 변화 (아침 밝은 카페, 밤 어두운 라운지 등).

**말풍선 리액션의 의미**: "대화 없는 소속감" thesis를 강화. 말을 할 수 없지만 존재를 인정하는 행위. 인사 = 환영, 축하 = 완료 시 축하, 파이팅 = 응원.

## DB Schema (초안)

```sql
-- Supabase anonymous auth가 auth.users 자동 생성
-- 아래는 public 스키마

CREATE TABLE sessions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES auth.users(id),
  task_name TEXT NOT NULL CHECK (char_length(task_name) <= 50),
  duration_minutes INT NOT NULL CHECK (duration_minutes IN (25, 50, 90)),
  started_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  status TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active', 'completed', 'timeout')),
  completed_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_sessions_user_id ON sessions(user_id);
CREATE INDEX idx_sessions_active ON sessions(status) WHERE status = 'active';
```

## Reviewer Concerns (3라운드 리뷰 후 미해결)

아래 항목은 3라운드 spec review에서 제기되었으나 MVP 범위를 넘어서거나 구현 시 결정해도 충분한 항목:

1. **에러 추적 미명시**: PostHog은 제품 분석용이며, 에러 트래킹(Sentry 등)과 Supabase 헬스체크 모니터링이 명시되지 않음. 구현 시 Sentry 무료 티어 추가 권장.
2. **남용 방지 미명시**: 익명 인증 + 레이트 리밋 없음 = 봇이 방을 가짜 세션으로 채울 수 있음. 구현 시 IP 기반 레이트 리밋 (분당 5회 세션 생성) 추가 권장.
3. **데이터 보존 정책**: 세션 레코드가 무한 축적됨. 쿠키 삭제한 익명 유저의 고아 레코드 잔류. MVP에서는 허용, Phase 2에서 90일 자동 삭제 정책 도입.
4. **개인정보보호법(PIPA)**: 익명 서비스라도 한국 공개 웹서비스는 개인정보처리방침 + 이용약관 필요. 런칭 전 작성 필수.
5. **DiceBear CDN 지연**: API 다운이 아닌 느린 응답(2-3초) 시 그리드에 빈 아바타 깜빡임 가능. 구현 시 `@dicebear/core` 클라이언트 라이브러리 사용 또는 Entry 화면에서 아바타 프리로드 권장.
6. **"카페 효과" 전달력**: 정적 그리드가 "타이머 앱"으로 보일 위험은 여전히 존재. MVP 검증 시 핵심 질문: "함께 있는 느낌이 드셨나요?" — 이 답변이 Phase 2 전환 여부를 결정.
7. **시장 데이터 정확성**: SpaceCloud 33조 원 수치는 스터디카페만이 아닌 전체 공간 거래 포함일 수 있음. 피치덱에 사용 시 출처 정밀 확인 필요.
